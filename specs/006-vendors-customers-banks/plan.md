# Implementation Plan: Master Data — Vendors / Commercial Customers / Bank Accounts (Spec 006)

**Branch**: `006-vendors-customers-banks` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/006-vendors-customers-banks/spec.md`

## Summary

Build 3 complete, net-new Accountant/Admin master-data pages on the approved Mohasebaty design system: `vendors.html` (الموردون), `commercial-customers.html` (عملاء الشركة), and `bank-accounts.html` (الحسابات البنكية والخزنة). Each follows the **same proven shape** as the existing `clients.html`: header + client selector, a 4-card KPI row, a filter bar, a multi-column records table with an action column, and an "add" modal authored in HTML. All UI reuses existing components — **zero new CSS** — with Arabic/RTL static demo data, no backend, and JavaScript used only to open/close modals. The 3 sidebar coming-soon items are converted to live links across every page, and `index.html` cards/counters are updated (25→28 screens, 12→9 coming-soon).

**Lesson carried from Specs 004–005:** the only class names used MUST already exist in `assets/css/output.css` (the Spec 004 rogue-class defect). `clients.html` is the canonical reference to copy markup from; a rogue-class sweep gates sign-off.

## Phase Execution Notes

### No New JS Required
- **Add-vendor / add-customer / add-account modals** → `initOverlays()` (`data-open="#id"` / `data-overlay` / `data-close`) ✅
- **Filter dropdowns** → `initFilterSelects()` (visual only) ✅
No additions to `main.js`.

### Zero-New-CSS Rule (hard requirement)
Every class used MUST already exist in `assets/css/output.css`. Copy component markup verbatim from `clients.html` (the closest structural twin); run `quickstart.md` check (j) (rogue-class sweep) before sign-off. A minimal inline `style=""` is acceptable for a one-off gap — never a new class. (Note: `.btn-rose` is defined in `input.css` but purged from `output.css`; do NOT use it — use `.btn-primary`/`.btn-outline`/`.btn-amber`.)

### Three near-identical pages
All 3 pages share the same skeleton (header + KPI row + filter bar + table + add modal). The implementer authors `vendors.html` first (fully), then derives `commercial-customers.html` and `bank-accounts.html` by swapping KPIs, columns, demo rows, and modal fields — keeping the shared shell identical. Commercial-customers reuses the same filter bar pattern (search/status/country/taxable optional). Bank-accounts' table covers all 3 account types with a `.chip` type badge.

### Sidebar Wiring Strategy
All pages share the identical 3 `is-soon` items: `الموردون` (under "البيانات الأساسية"/master-data section), `العملاء التجاريون`, and `الحسابات البنكية` (under "البنوك"). A single Python script replaces all 3 across the 25 existing pages (same pattern as Specs 003–005). The 3 new pages are authored with live links and mark their own item `active` + `aria-current="page"`.

### index.html
- 3 landing cards (الموردون ~line 384, العملاء التجاريون ~391, الحسابات البنكية ~412): `<div class="landing-card is-soon">` → live `<a class="landing-card" href="pages/…html">` with a `.go` affordance.
- Counters: chip `12 وحدة` → `9 وحدة`; footer `25 شاشة مكتملة + 12 وحدة قادمة` → `28 شاشة مكتملة + 9 وحدة قادمة`.
