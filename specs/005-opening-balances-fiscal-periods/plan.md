# Implementation Plan: Opening Balances & Fiscal Periods (Spec 005)

**Branch**: `005-opening-balances-fiscal-periods` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/005-opening-balances-fiscal-periods/spec.md`

## Summary

Build 2 complete, net-new Accountant/Admin frontend pages on the approved Mohasebaty design system: `opening-balances.html` (client + go-live date, 4 balance status cards, a 7-column debit/credit table, validation card, add-balance modal, and upload/approve/export actions) and `fiscal-periods.html` (fiscal-year setup, 12 monthly periods with status + readiness counts, plus close-period and reopen-period modals). All UI reuses existing components ‚Äî **zero new CSS** ‚Äî and is Arabic/RTL with static demo data only (no backend, no real arithmetic). The 2 sidebar coming-soon items are converted to live links across every page, `index.html` cards/counters are updated (23‚Üí25 screens, 14‚Üí12 coming-soon), and contextual links are added from `settings.html`, `reports.html`, and `chart-of-accounts.html`.

**Lesson carried from Spec 004:** the 004 verification caught pages using *rogue* CSS class names absent from `output.css` (rendering off-design). This plan therefore mandates that **every class used must already exist in `assets/css/output.css`** (catalog confirmed below); the implementer copies markup from the named reference pages rather than inventing child classes.

### Zero-New-CSS Rule (hard requirement)
Every class used MUST already exist in `assets/css/output.css`. The 004 verification found rogue child classes (`.tc-ico`, `.dz-title`, `.fc-name`, `.steps-list`‚Ä¶) that were undefined and rendered off-design. To prevent recurrence: copy component markup verbatim from the reference page named in the reuse map, and run `quickstart.md` check (j) (rogue-class sweep) before sign-off. If a true structural gap appears, use a minimal inline `style=""` (the `client-tax-settings.html` precedent) ‚Äî never a new class.
