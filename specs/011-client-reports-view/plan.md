# Implementation Plan: Client Reports View (Spec 011)

**Branch**: `011-client-reports-view` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/011-client-reports-view/spec.md`

## Summary

Build one complete, net-new **client-facing** page on the approved Mohasebaty design system: `client-reports.html` (تقاريري — a friendly header with تحميل آخر تقرير + طلب تقرير, 3 simple filters, 5 summary cards, a 5-column reports table, plain-language status explanations, a tax-estimate note, and a request-report card). All UI reuses existing components — **zero new CSS** — Arabic/RTL, static data, no backend/enforcement. This is a **client portal** page (Principle V): simple language, and it MUST NOT contain any internal accounting terminology (no مدين/دائن، قيد، ترحيل، شجرة الحسابات، account codes, internal notes). The last sidebar coming-soon item is converted to a live link across every page, `index.html` is updated so **no coming-soon module remains**, and contextual links are added from `client-dashboard.html` and `client-documents.html`.

**Lessons carried from Specs 004–010 (hard rules):** every class used MUST already exist in `assets/css/output.css`; **copy KPI / filter / form markup verbatim** from the named reference pages (use the real `.kpi-num`/`.kpi-lbl`/`.kpi-help`; the amber chip is `.chip.amber` not `.chip-amber`); **do NOT use `.btn-rose`** (purged → `.btn-amber`/`.btn-outline`). A rogue-class sweep gates sign-off. **Sidebar wiring uses the label-based line fixer** (locate the label line, rewrite its immediate wrapper) — NEVER a spanning/DOTALL regex; a balanced-sidebar check guards against recurrence. **NEW for this feature:** a client-internals keyword scan gates the page (SC-004).
