# Implementation Plan: Modals & Cross-Page Actions Completion (Spec 012)

**Branch**: `012-modals-cross-page-actions` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/012-modals-cross-page-actions/spec.md`

## Summary

Close the prototype's "dead action" gap before backend work. Across the existing **37 pages**, every major workflow action (إضافة، فتح، عرض، مراجعة، تعديل، اعتماد، رفض، إغلاق، إعادة فتح، تأكيد الترحيل، تصدير، رفع، إسناد) must resolve to one honest outcome: **navigate** to a real page, **open a modal**, or render as a **clearly disabled/"قريباً"** control. Of the 16 named modals, **9 already exist and ship today** (Add Client/Vendor/Commercial Customer/Bank Account/Document Category/User, Add Manual Journal Entry, Close & Reopen Fiscal Period); this feature adds the **7 missing modals** (Request Clarification, Reject Document, Approve Report, Confirm Posting, Upload Data File, Import Error Action, Assign Accountant), plus one reusable **Export confirmation** pattern, then sweeps the ~99 `href="#"` placeholders so no major action is dead.

**The defining constraint: zero new CSS and zero new JS.** The shared modal system already exists in `output.css` (`.scrim`/`.modal`/`.modal-head`/`.modal-body`/`.modal-foot`) and the shared `initOverlays()` in `main.js` already wires **any** `[data-open]`/`[data-close]`/scrim-click/Escape at boot. Every new modal is authored as **static HTML** that reuses confirmed classes; every new trigger is a `data-open="#id"` attribute. Nothing is generated by JS (FR-009). No Tailwind rebuild is required.

**Lessons carried from Specs 004–011 (hard rules):** every class used MUST already exist in `assets/css/output.css` — a rogue-class sweep gates sign-off. **Do NOT invent a danger/red button**: the approved palette has only `.btn-primary` / `.btn-outline` / `.btn-amber` / `.btn-ghost` (`.btn-rose`/`.btn-danger` were purged). Map severity to existing classes: warning/destructive confirm → `.btn-amber`, positive confirm → `.btn-primary`, cancel → `.btn-outline`. **Sidebar/shared markup is never touched by a spanning regex.** **Do not modify the 9 already-shipping modals** — reuse and align only (Principle VII). **NEW for this feature:** an action-coverage audit (no dead major action) and a client-boundary check on every new modal gate sign-off.

## Phase Execution Notes

### Zero-New-CSS / Zero-New-JS (hard requirement)
Every class used MUST already exist in `output.css`; every modal opens via existing `data-open`/`initOverlays()`. No `input.css` edit, no `output.css` rebuild, no `main.js` edit. Verified by `quickstart.md` rogue-class sweep + a "no `main.js` diff" check.

### Severity → existing button class (no danger/red invented)
- Positive confirm (Approve Report, Confirm Posting, Reopen Period, all create/upload) → `.btn-primary`
- Warning/destructive confirm (Request Clarification, Reject Document; Close Period already shipped) → `.btn-amber`
- Cancel → `.btn-outline[data-close]`; Close X → `.icon-btn[data-close]`

### Client-portal boundary (defining gate, Principle V)
Approve Report attaches to **accountant** `reports.html`/`approval-workflow.html`, never to client `client-reports.html`. Clarification/Reject/Confirm-Posting/Import-Error are accountant-only. Any client-facing modal (e.g. an export confirmation on `client-reports.html` / `client-dashboard.html`) stays simple — a client-internals keyword scan (مدين/دائن/قيد/ترحيل/شجرة الحسابات/account codes) MUST return zero hits in client-facing modal bodies.

### Cross-page dead-action sweep (the ~99 `href="#"`)
Apply the decision taxonomy from `contracts/page-contracts.md#C3` to every major action:
- **عرض/فتح/مراجعة** → real existing page (e.g. document-review.html, client-document-details.html, client-profile.html).
- **إضافة/إسناد** → `data-open="#…Modal"` (existing or new).
- **تعديل** → reuse the entity's add modal as edit (open same modal) or navigate to its detail page.
- **اعتماد/رفض/تأكيد الترحيل/إغلاق/إعادة فتح** → confirmation modal.
- **تصدير** → `data-open="#exportConfirmModal"`.
- **رفع** → upload modal or upload page.
- **Truly future** → disabled/"قريباً" control (no active-looking `#`).
- **JS-handled** `href="#"` (dropdown/user-menu/tab/notification toggles) → leave; verify it produces a visible response (already wired) so it is not "dead".
Minor links (pagination, sort) must not look broken but are not forced to navigate.

### Do not touch the 9 shipped modals
`addClientModal`, `addVendorModal`, `addCustomerModal`, `addAccountModal`, `addCategoryModal`, `addUserModal`, `manualEntryModal`, `closePeriodModal`, `reopenPeriodModal` are reused as-is. New triggers may point at them, but their markup is not edited.
