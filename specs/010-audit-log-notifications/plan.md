# Implementation Plan: Audit Log & Notifications Center (Spec 010)

**Branch**: `010-audit-log-notifications` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/010-audit-log-notifications/spec.md`

## Summary

Build 2 complete, net-new Admin/Accounting pages on the approved Mohasebaty design system: `audit-log.html` (سجل النشاط — 5 filters, a 4-card KPI band, a 9-column event table, and a before/after side details panel) and `notifications.html` (مركز الإشعارات — a 6-chip filter row, a 6-item notification list with status/actions, and a 4-toggle preferences card). All UI reuses existing components — **zero new CSS** — Arabic/RTL, static data, no backend/enforcement. The 2 sidebar coming-soon items are converted to live links across every page, `index.html` cards/counters are updated (34→36 screens, 3→1 coming-soon), the header notification **bell** dropdown's "عرض الكل" footer is wired to `notifications.html` (non-destructively, preserving the dropdown), and contextual links are added from `settings.html`.

**Lessons carried from Specs 004–009 (hard rules):** every class used MUST already exist in `assets/css/output.css`; **copy KPI / filter / notification / bell markup verbatim** from the named reference pages (several inner classes like `.kpi-val`/`.notif-list` do NOT exist — the real ones are `.kpi-num`/`.kpi-lbl`/`.notif-item`); **do NOT use `.btn-rose`** (purged → `.btn-amber`/`.btn-outline`). A rogue-class sweep gates sign-off. **Sidebar wiring uses the label-based line fixer** (locate the label line, rewrite its immediate wrapper) — NEVER a spanning/DOTALL regex (that bug scrambled the sidebar in Spec 008 and had to be reverted); a balanced-sidebar check guards against recurrence.

## Phase Execution Notes

### No New JS Required
- **Filter chips** (notifications) → `initSingleSelect()` ✅; **filter dropdowns** (audit-log) → `initFilterSelects()` ✅
- **Switches** (preferences) → static `<input type="checkbox">` inside `.switch` ✅
- **Header bell dropdown** → existing bell-toggle behavior is untouched; only the footer link's `href` changes ✅
No additions to `main.js`.

### Zero-New-CSS Rule (hard requirement)
Every class used MUST already exist in `output.css`. **Copy KPI/filter/notification/bell markup verbatim** from the reference pages — do not guess inner class names (`.kpi-num`/`.kpi-lbl`/`.kpi-help` exist; `.kpi-val`/`.kpi-label` do NOT; `.notif-item` exists; `.notif-list` does NOT). **Do NOT use `.btn-rose`.** Verified by `quickstart.md` check (j).

### Header-Bell Wiring (non-destructive)
The bell is an interactive dropdown (`data-bell-toggle` + `.notif-panel`) on 11 pages. **Preserve it.** For each `.notif-panel`, ensure its footer has a link to `notifications.html` ("عرض كل الإشعارات ←"): where a footer `<a href="#">عرض الكل …</a>` already exists in the panel, repoint its `href`; otherwise insert one immediately before the panel's closing `</div>`. Reuse the existing in-panel link styling (descendant `.notif-panel a`). Do NOT alter the bell button or the dropdown items.

### Sidebar / index / Contextual Links
- Convert the 2 `is-soon` spans (`سجل التدقيق` → `audit-log.html`; `الإشعارات` → `notifications.html`) across all existing pages using the **label-based line fixer** (NEVER a spanning regex). New pages mark their own item `active` + `aria-current="page"`. After this feature only `تقارير العميل` remains coming-soon.
- `index.html`: convert the 2 `is-soon` landing cards (سجل التدقيق ~line 447, الإشعارات ~line 461) to live `<a class="landing-card">`; chip `3 وحدة`→`1 وحدة`; footer `34 شاشة مكتملة + 3 وحدة قادمة` → `36 شاشة مكتملة + 1 وحدة قادمة`.
- Contextual links: `settings.html` → `audit-log.html` ("سجل النشاط") and → `notifications.html` ("مركز الإشعارات").
